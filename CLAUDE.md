# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

All tasks run through Nx. Prefix with `npx` (or the workspace package manager):

```sh
# Build a package
npx nx build @cdk-x/core
npx nx build hetzner          # short name works too

# Test a package
npx nx test @cdk-x/core
npx nx test hetzner-runtime

# Run a single test file
npx nx test @cdk-x/core --testFile=packages/core/src/lib/stack/stack.spec.ts

# Lint
npx nx lint @cdk-x/core

# Format (check / fix)
npx nx run @cdk-x/hetzner:format:check
npx nx run @cdk-x/hetzner:format

# Build all
npx nx run-many -t build

# Run affected tasks only (CI-style)
npx nx affected -t build,test,lint

# Regenerate Hetzner L1 constructs from JSON Schema
npx nx run @cdk-x/hetzner:codegen

# Run the CLI locally
npm run cdkx -- synth --app "npx ts-node my-app.ts"
```

## Architecture

**cdkx** is a multi-provider CDK-like framework. Users write TypeScript construct trees; cdkx synthesizes provider-specific JSON/YAML manifests, then deploys them via provider HTTP APIs.

Two phases:
1. **Synth** (`cdkx synth`) ‚Äî runs the user's app, produces a cloud assembly (`cdkx.out/manifest.json` + one `<StackId>.json` per stack).
2. **Deploy** (`cdkx deploy`) ‚Äî reads the cloud assembly, resolves `{ ref, attr }` tokens, builds a dependency DAG, calls the provider API in topological order.

### Package map

| Package | Path | Role |
|---|---|---|
| `@cdk-x/core` | `packages/core/` | Construct primitives (`App`, `Stack`, `ProviderResource`), synthesis pipeline, cloud assembly format, `ResolverPipeline`, `Lazy`, runtime base classes |
| `@cdk-x/engine` | `packages/engine/` | Deployment runtime ‚Äî reads manifests, resolves tokens, drives the deploy state machine |
| `@cdk-x/cli` | `packages/cli/` | CLI binary (`cdkx synth/deploy/destroy`) ‚Äî imports engine |
| `@cdk-x/hetzner` | `packages/providers/hetzner/hetzner/` | Hetzner Cloud provider + auto-generated L1 constructs |
| `@cdk-x/hetzner-runtime` | `packages/providers/hetzner/hetzner-runtime/` | Hetzner deployment handlers (`ResourceHandler` subclasses) + `HetznerRuntimeAdapterFactory` |
| `@cdk-x/hetzner-sdk` | `packages/providers/hetzner/hetzner-sdk/` | Raw Hetzner Cloud API client |
| `@cdk-x/testing` | `packages/testing/` | Shared test helpers: `TestApp`, `TestStack`, `SynthHelpers` |
| `@cdk-x/logger` | `packages/logger/` | Logger interface |
| `@cdk-x/spec-to-cdkx` | `packages/tools/spec-to-cdkx/` | Code generator: JSON Schema ‚Üí TypeScript L1 constructs |

### Construct tree

```
App (root)
 ‚îî‚îÄ‚îÄ Stack (one per provider / deployment unit)
      ‚îî‚îÄ‚îÄ ProviderResource (one per resource ‚Äî base for L1 NtvXxx classes)
```

- `App.of(construct)` / `Stack.of(construct)` walk up the tree to find the root.
- `Stack` takes an optional `StackProps` (`synthesizer`, `description`, `stackName`); no provider prop ‚Äî each resource carries its own provider implicitly.
- `ProviderResource.logicalId` is a stable `<humanPrefix><8-char-SHA256>` identifier. Used as the JSON key in the stack template.
- L1 classes use a short provider prefix set at codegen time (e.g., `Htz` for Hetzner ‚Üí `HtzNetwork`, `HtzSubnet`). Auto-generated by `spec-to-cdkx` via `--prefix`.

### Synthesis pipeline

`app.synth()` ‚Üí for each Stack: calls `synthesizer.synthesize()` ‚Üí for each `ProviderResource`: calls `resource.toJson()` ‚Üí runs `ResolverPipeline.resolve()` recursively over all properties ‚Üí writes `<artifactId>.json` ‚Üí writes `manifest.json`.

**ResolverPipeline order:** global resolvers ‚Üí provider resolvers ‚Üí `LazyResolver` ‚Üí `ImplicitTokenResolver`.

**Cross-resource references:** `resource.getAtt('attrName')` returns an `IResolvable` that serializes to `{ "ref": "<logicalId>", "attr": "<attrName>" }`. The engine reads these tokens to build the dependency graph and resolves them after the dependency is created.

### Deployment engine (engine package)

```
CloudAssemblyReader ‚Üí DeploymentPlanner (builds DAG) ‚Üí DeploymentEngine (state machine)
  ‚îî‚îÄ‚îÄ ProviderAdapter (interface)
       ‚îî‚îÄ‚îÄ RuntimeAdapter (bridges engine ‚Üî ProviderRuntime/ResourceHandler)
            ‚îî‚îÄ‚îÄ HetznerRuntimeAdapterFactory (creates the Hetzner RuntimeAdapter from env vars)
```

Resource lifecycle states: `PENDING ‚Üí CREATING ‚Üí CREATED | FAILED`, `UPDATING`, `DELETING`. Resources process in topological order; failed resources block dependents.

### Adding a new resource handler (Hetzner example)

1. Add JSON Schema file for the resource type under `packages/providers/hetzner/hetzner/`.
2. Run `npx nx run @cdk-x/hetzner:codegen` to regenerate `resources.generated.ts` and `runtime-config.generated.ts`.
3. Create a `XxxHandler extends ResourceHandler<TProps, TState, HetznerSdk>` in `packages/providers/hetzner/hetzner-runtime/src/lib/handlers/`.
4. Register it in `HetznerProviderRuntime` constructor with `this.register('Hetzner::Domain::Type', new XxxHandler())`.

## Design rules

- **Everything is a class.** No standalone exported functions ‚Äî use static or instance methods on classes.
- **No `any`** except `Lazy.any()` (intentional escape hatch; has `eslint-disable` comment).
- **CommonJS throughout.** No `"type": "module"`. Local imports are extensionless.
- **Circular dependencies:** resolve with lazy `require()` inside the function body (established pattern in `ProviderResource.toJson()`).
- **camelCase props, snake_case SDK calls.** Handlers map explicitly between the two.
- **Structured logging.** Pass a string event key and a context object: `ctx.logger.info('provider.handler.network.create', { name })`.

## Conventions

### Session conventions

- **Commits are never made automatically.** Only create a commit when the user explicitly asks for it.
- **Format + test before committing.** When the user asks to commit, run the formatter and tests first (unless already done in the same session step).
- **English only.** All documentation (markdown files), JSDoc comments, and inline code comments must be written in English.

### Coding conventions

| Rule | Detail |
|------|--------|
| Everything OOP | No standalone `export function`. All utilities are static methods on classes. |
| No `any` | Use `unknown` everywhere. The one exception is `Lazy.any()` return type ‚Äî intentional escape hatch, gets `eslint-disable` comment. |
| CJS imports | All local imports use **no file extension** (extensionless). `moduleResolution: node` resolves them correctly. |
| Unused params | ESLint's `argsIgnorePattern: "^_"` does NOT suppress warnings for class method params. Fix: **omit the parameter entirely** from the method signature. TypeScript allows implementing an interface method with fewer params than declared. |
| Prettier | Run `npx nx run <package>:format` after writing or modifying any `.ts` file. Config: `singleQuote`, `trailingComma: all`, `printWidth: 80`, `tabWidth: 2`, `semi: true`. |
| Specs co-located | `foo/foo.spec.ts` lives next to `foo/foo.ts`. |
| Module format | CommonJS throughout ‚Äî no `"type": "module"` in any `package.json`. |

### New library checklist

When creating a new library, follow these steps in order:

1. **Generate with Nx** ‚Äî use `nx-generate` skill. Always dry-run first (`--dry-run`). Preferred command:
   ```bash
   npx nx g @nx/js:library packages/<name> \
     --name=<name> --importPath=@cdk-x/<name> \
     --bundler=tsc --publishable --unitTestRunner=jest \
     --linter=eslint --minimal --useProjectJson --no-interactive
   ```

2. **Align `project.json`** ‚Äî replace generated targets with workspace standard: `build` using `@nx/js:tsc`, `format`/`format:check` using prettier (same pattern as `@cdk-x/core`), `test` with `passWithNoTests: true`.

3. **Align `package.json`** ‚Äî `"version": "0.1.0"`, remove `"type": "module"`, `exports` must use `"require"` (not `"import"`), add `"publishConfig": { "access": "public" }`, use `"tslib": "^2.8.1"` in dependencies.

4. **Align `tsconfig.lib.json`** ‚Äî remove `"module": "nodenext"` and `"moduleResolution": "nodenext"`, add `"declaration": true` and `"declarationMap": true`, add `"test/**/*"` to the `exclude` list.

5. **Align `tsconfig.spec.json`** ‚Äî remove `"module": "nodenext"` and `"moduleResolution": "nodenext"`, add `"test/**/*.ts"` to `include`.

6. **Align `jest.config.cts`** ‚Äî set `displayName` to the full package name (e.g. `'@cdk-x/engine'`), remove the `/* eslint-disable */` header comment.

7. **Clean up generated stubs** ‚Äî delete `src/lib/<name>.ts` and `src/lib/<name>.spec.ts`. Replace `src/index.ts` content with a simple comment placeholder.

8. **Add to `nx.json` release group** ‚Äî add the new package to the appropriate release group's `projects` array (typically `core`).

9. **Add scope to `.vscode/settings.json`** ‚Äî add the package name to the `"conventionalCommits.scopes"` array.

10. **Create `AI.md`** ‚Äî add an `AI.md` at the package root documenting what the package does, its role, architecture, and relationship with other packages.

11. **Add to `opencode.json` instructions** ‚Äî add the new package's `packages/<name>/AI.md` to the `instructions` array.
